# V2.1.0 Implementation Spec — Workflow / Sub-agent Attribution

> **How to use this document:** open a fresh Claude Code session on a clean
> `feature/workflow-attribution` branch (cut from `main` after v2.0.2 merges)
> and hand it this file with: *"Implement docs/V2.1-WORKFLOW-SPEC.md. Work
> phase by phase; compile after each phase; stop and report after Phase 2 and
> after Phase 4 for checkpoint reviews."* Everything an implementer needs is
> in here — the why, the data formats, the file map, acceptance criteria, and
> known traps.

---

## 1. Goal and product framing

Claude Code's dynamic workflows (trigger word **ultracode**) dispatch tens to
hundreds of background sub-agents. Each sub-agent writes its own usage log.
**Today the extension reads those logs but cannot attribute them**: a single
observed ultracode run produced 879 sub-agent records / 24.6M tokens versus
77 records / 4.6M tokens in the main session — the sub-agent share (~84%) is
invisible to the user as a *workflow*, and was historically miscounted.

v2.1.0 makes workflow consumption a first-class, visible thing, in line with
the product positioning (see `CLAUDE.md`): **token precision over cost
precision; help the user decide what's worth using**. Keep every UI addition
minimal — workflow-level summaries first, drill-down on demand.

A second pillar (added 2026-06-12): a **usage attribution panel** modelled on
the official Claude Code `/usage` screen ("What's contributing to your limits
usage?"), with two deliberate differences from the official one:
(a) it covers **all models/providers** seen in the logs, not just Claude, and
(b) it offers **finer scopes** — Day / Week / **Month / per-Session /
per-Project** instead of Day/Week only — plus extra classifications the
official screen lacks (workflow share, subagent types beyond workflows).
A compact summary lands in the Today tab; the full breakdown and the
smarter AI advice land in the Content tab and the advisor (Phases 4–5).

Non-goals for v2.1.0: multi-provider dashboards, billing reconciliation,
historical data migration, settings webview.

## 2. Data formats (verified on disk, 2026-06)

### 2.1 Directory layout

```
~/.claude/projects/
└── <proj-encoded>/                          e.g. d--Jiaming-PolyU-research
    ├── <session-id>.jsonl                   main conversation
    └── <session-id>/                        per-session asset dir
        ├── subagents/
        │   ├── agent-<hex>.jsonl            generic sub-agent (Task tool)  ★
        │   ├── agent-<hex>.meta.json        {"agentType": "Explore" | …}
        │   └── workflows/
        │       └── wf_<id>/                 one dir per workflow run
        │           ├── agent-<hex>.jsonl    per-sub-agent usage log  ★
        │           ├── agent-<hex>.meta.json  {"agentType":"workflow-subagent"}
        │           └── journal.jsonl        workflow journal (no usage)
        ├── workflows/
        │   ├── wf_<id>.json                 workflow definition  ★
        │   └── scripts/<name>-wf_<id>.js    generated script
        └── tool-results/…                   misc artifacts (ignore)
```

★ = required inputs for this feature.

### 2.2 `agent-*.jsonl` records

Same line format as the main conversation log: each usage line has
`timestamp`, `message.usage.{input_tokens, output_tokens,
cache_creation_input_tokens, cache_read_input_tokens}`, `message.model`,
`message.id`, and usually `cwd`. Verified: agent records carry **real token
values** (no zero-placeholder problem) for both native Claude and CC Switch /
DeepSeek routing. `journal.jsonl` lines have **no** `message.usage` — the
loader's validator already rejects them; do not special-case.

**Trap:** sub-agents spawned with worktree isolation have a `cwd` pointing at
a temporary git worktree, *not* the workspace folder. Workspace filtering by
cwd therefore misses them. The loader (since v2.0.2) resolves the parent
session and encoded project dir from the *file path* (`parseSessionInfo`), so
**prefer path-derived attribution over cwd for workflow records**.

**`agent-*.meta.json`** (verified 2026-06-12, 93 files on this machine):
single JSON object `{"agentType": "<type>"}`. Observed values:
`workflow-subagent` (44), `Explore` (24), `general-purpose` (21),
`claude-code-guide` (3), `Plan` (1). Generic sub-agents live directly under
`subagents/`; workflow sub-agents under `subagents/workflows/wf_<id>/`.
Read defensively — missing/unparseable meta means agentType "unknown".

### 2.3 `wf_<id>.json` (workflow definition)

Read defensively (shape is not a stable API): expect at least an id; a
human-readable name may exist in the JSON or can be derived from the script
filename `scripts/<human-name>-wf_<id>.js` (e.g.
`deep-literature-review-wf_fcfc35cc-5d5.js` → "deep-literature-review").
Fallback display name: the `wf_<id>` string.

### 2.4 Loader state you can rely on (already shipped in v2.0.2)

- `findJsonlFiles` already walks recursively → agent files are already read.
- `parseSessionInfo(filePath)` already maps nested files to the parent
  `sessionId` + encoded project dir.
- Records carry `_sessionId`, `_projectPath`, `_projectName`,
  `_sessionTitle?`, `_gitBranch?`.
- Diagnostic logging (`options.log`) prints per-refresh stats + models seen.

### Phase 0 — AI-advice reliability (bug fix, ships even if later phases slip)

**Problem:** `Get AI Usage Advice` intermittently fails with `terminated` or
hangs at the progress notification (v2.0.0 fine; regression noticed during
v2.0.2 testing). Root-cause hypotheses, in order: (a) network instability on
Node fetch to `api.deepseek.com` (same class of problem as the Anthropic
TLS-fingerprint issue solved in `claudeApiClient.ts` by shelling out to
`curl`); (b) oversized request body — `buildAdviceSummary` includes up to 80
prompts × 2500 chars.

**Changes in `src/advisor.ts` (+ `src/extension.ts` where noted):**
1. Add an `AbortController` timeout (default 120 s) around the fetch; on
   abort, surface a clear "timed out" error instead of hanging forever.
2. On fetch failure (network error, `terminated`, abort), retry once; if the
   retry fails, fall back to the same `curl` transport pattern used in
   `claudeApiClient.ts` (factor the curl helper out of `claudeApiClient.ts`
   into a small shared `src/httpClient.ts` rather than duplicating it).
3. Cap the summary payload: at most 40 prompts × 1500 chars each, and skip
   the prompts section entirely above ~80 KB total.
4. **User-message purity:** advice prompts must contain only top-level user
   messages from main sessions. In the content analyzer's prompt collection
   (`collectPrompt` in `dataLoader.ts`), skip records whose source file lies
   under a `subagents/` directory (the loader knows the file path at parse
   time — thread a flag through), and skip obvious agent-framework text
   (lines starting with structured scaffolding, e.g. XML-ish tags).

**Acceptance:** advice runs 5/5 times without hanging on a flaky network
(simulate by pointing `advice.apiUrl` at a black-holed host — expect a clean
timeout error, not a hang); prompts sample contains no sub-agent content.

### Phase 1 — Workflow data model in the loader

**`src/types.ts`:** add

```ts
export interface WorkflowUsage {
  workflowId: string;          // "wf_fcfc35cc-5d5"
  name: string;                // derived human name or the id
  sessionId: string;           // parent session
  projectPath: string;         // encoded project dir or cwd-derived path
  projectName: string;
  startTime: Date;             // min record timestamp across agents
  endTime: Date;               // max record timestamp
  agentCount: number;
  data: UsageData;             // aggregated across all agent files
  agents: { agentId: string; data: UsageData; startTime: Date; endTime: Date }[];
}
```

**`src/dataLoader.ts`:**
1. In the per-file loop, when a file path contains `/subagents/`, extract
   `_agentId` (file basename) and `_agentType` (from the sibling
   `agent-<hex>.meta.json`, cached per refresh; "unknown" on failure) onto
   each record; when the path additionally contains `/subagents/workflows/`,
   also extract `_workflowId` (the `wf_*` segment). Add all three as
   optional fields on `ClaudeUsageRecord`.
2. New `static getWorkflowBreakdown(records, limit = 50): WorkflowUsage[]` —
   group by `_workflowId`, aggregate with the existing `calculateUsageData`,
   sort by `endTime` desc. Resolve `name` by checking (cheap, cached per
   refresh) the `workflows/scripts/` directory next to the agent dir for a
   `<name>-wf_<id>.js` file; fall back to the id.
3. Main-session records (no `_workflowId`) are untouched.

**Acceptance:** a quick `node -e` harness against the real
`d--Jiaming-PolyU-research` data shows ≥2 workflows with human-readable
names, 31 agents between them. **Token-count note (verified 2026-06-12):**
the raw line sum over those agent files is 879 usage lines / 24.6M tokens,
but every `message.id`+`requestId` pair there appears 2–3× (the proxy
placeholder+final double-write of issue #18), so the loader's dedup
correctly reports ~314 requests / ~8.9M tokens. The deduped figure is the
dashboard-consistent one — do not chase the raw 24.6M.

### Phase 2 — Workflows tab (webview)

New tab "Workflows" after "Branches" (`I18n` keys for all six languages —
en "Workflows", zh-CN "工作流", zh-TW "工作流", ja "ワークフロー",
ko "워크플로", de "Workflows").

Layout per the project's minimal style (reuse `.daily-table sortable-table`
and the collapsible `<details>` pattern from Model Usage):

- One row per workflow: start time · name · project · **agent count** ·
  cost · input / output / cache-write / cache-read · **cache hit rate** ·
  duration. Sortable columns via the existing `data-sort-*` mechanism.
- Row expands (`<details>` or the project-group toggle pattern) to a
  per-agent sub-table: agent id (short), cost, tokens, duration.
- A one-line summary strip above the table: workflows this month · total
  workflow cost · share of this month's total cost.
- **Cache hit rate is the headline diagnostic**: per workflow, `cacheRead /
  (input + cacheWrite + cacheRead)`. Render it with the existing summary
  conventions. Rationale (documented in a hint line under the table):
  native-Claude workflows reuse prompt cache across agents (high hit rate);
  a provider with no cross-agent cache shows ~0% — i.e. the same workflow
  costs disproportionately more. This is the "is my proxy well-adapted to
  workflows" signal.

**Acceptance:** tab renders with real data; expanding a workflow lists its
agents; sorting works; no layout overflow at 800 px container width.

### Phase 3 — Thinking share + quota guard

1. **Thinking share column** (Sessions tab) and a line in the Today summary:
   thinking tokens are already estimated by the content analyzer
   (`catAssistantThinking`); expose a per-session share where the data
   exists. Mark as estimate (reuse `estimatedNote` styling).
2. **Effort hint:** if a session's thinking share > 60%, show an inline hint
   ("Consider /effort high instead of xhigh for tasks like this") in the
   Sessions row tooltip — *no popups*.
3. **Quota guard (the high-value one):** when the dashboard renders and the
   5h quota's remaining percentage is **below the average cost of the user's
   last 3 workflow runs** (as % of the 5h window — derivable once Phase 1
   lands by correlating workflow cost share against quota utilisation deltas;
   if that's too indirect, fall back to a fixed configurable threshold
   `claudeCodeUsage.workflowQuotaWarnPercent`, default 50), show a dismissible
   warning banner in the dashboard: "A typical workflow run uses ~N% of your
   5-hour window; you have M% left. Consider waiting for the reset —
   interrupted runs lose their prompt cache and re-run ~40% more expensive."
   Status bar stays untouched (no nagging).

**Acceptance:** thinking column renders; banner appears when (and only when)
remaining 5h % < threshold; banner is dismissible per window-session.

### Phase 4 — Usage attribution panel ("What's contributing to your usage?")

Modelled on the official `/usage` screen, but multi-provider and with finer
scopes. Two render sites: **full panel in the Content tab**, **compact
summary in the Today tab**. ⛔ checkpoint: stop and report after this phase.

**4a. Data layer (`dataLoader.ts`)** — new
`static getUsageAttribution(records, scope): UsageAttribution`.
*Weight metric:* estimated cost per record (the dashboard's existing cost
model); fall back to total tokens when a model has no pricing. All shares
are weight-of-scope fractions.

- **Scopes:** `day` (today), `week` (last 7 d), `month` (last 30 d), plus
  `session:<id>` and `project:<path>` (same computation filtered to one
  session / project). The official screen only has Day/Week.
- **Characteristics** (independent signals, *not* a breakdown — mirror the
  official disclaimer wording):
  1. share of usage at **>150k context** — per record, context size =
     `input + cache_read + cache_creation`; hint: "/compact mid-task,
     /clear when switching tasks".
  2. share from **long-running sessions** — sessions with ≥8 *distinct
     active hours* (hour buckets containing ≥1 record; robust to idle
     gaps); hint: often background/loop sessions, make sure intentional.
  3. share from **subagent-heavy sessions** — sessions where records under
     `/subagents/` carry >50% of the session's weight; hint: each subagent
     runs its own requests, consider a cheaper model for simple ones.
  4. share from **workflow runs** (ours, not in the official screen) — all
     records with `_workflowId`; hint: see Workflows tab; ties into the
     Phase 3 quota guard.
- **Tables** (each hidden when empty):
  - **Skills** — detect skill invocations in assistant `tool_use` blocks
    named `Skill` (input carries the skill name) and `<command-name>`
    markers in user-visible text. **Share methodology (matched to the
    official screen, verified 2026-06-12):** a skill's share is the
    cost-weight of all scoped records in the same session *at or after*
    the skill's (earliest) invocation — i.e. "usage that came from this
    skill being active", overlapping is fine because these are
    independent characteristics. The injected-prompt token estimate
    (tool-result text length) is kept as a secondary column only — it
    measures ~0% and must not be used as the share.
  - **Subagents** — group subagent-record weight by `_agentType`
    (workflow-subagent / Explore / general-purpose / …).
  - **Plugins** — skills whose name has a `plugin:` namespace prefix
    (e.g. `frontend-design:frontend-design`) grouped by prefix; share =
    per-session earliest invocation among the plugin's skills, then the
    same at-or-after rule (no double counting within one plugin).
  - **Models** (ours; the official screen is Claude-only) — weight share by
    model, so a mixed Claude + DeepSeek user sees the provider split here.
- **Skill/plugin characteristic lines** (official parity): when the top
  skill's share ≥10%, add a line "{pct}% of your usage came from {skill}"
  with the hint "Heavy skills can be scoped down or run with a cheaper
  model via skill frontmatter."; same for the top plugin with "its agents,
  skills and MCP tools all count toward your limit." The disclaimer line
  includes "does not include other devices or claude.ai".

**4b. Content tab UI** — a "Usage attribution" section: scope selector
(Day / Week / Month / Session / Project; Session and Project reveal a
second `<select>` of recent sessions / known projects), the official-style
disclaimer line ("Approximate, based on local sessions on this machine —
independent characteristics, not a breakdown"), characteristic lines with
their hints, then the four sub-tables. Reuse `.daily-table` + `<details>`
patterns; everything marked approximate (`estimatedNote` styling).

**4c. Today tab summary** — a compact strip: only the characteristic lines
whose share ≥5% for today, one line each (percentage + short label, no
hint paragraphs), with a "details → Content tab" pointer. Renders nothing
when no line crosses 5% — zero noise for light days.

**Acceptance:** panel renders on real data in all five scopes; skills
(superpowers / context7 / …) appear with plausible counts; subagent table
shows ≥2 agent types on this machine's data; zero-skill/zero-subagent users
see those tables hidden; Today strip only shows ≥5% lines; no layout
overflow at 800 px.

## 7. Verified log-format facts (probed on disk 2026-06-13 — do not re-derive)

These answer the questions that motivated Part II. A probe script lives in the
git history of this branch (`out/_logprobe*.js`, gitignored); the conclusions:

1. **Effort is NOT in the logs.** There is no `effort` / `ultracode` /
   `maxThinking` / reasoning-budget field on any usage line. The only mode
   record is a `type:"mode"` line carrying `permissionMode` (e.g. `"normal"`)
   — nothing about ultracode/xhigh. **We cannot read the effort level a run was
   launched with.** (Trap: a naive substring grep for `effort`/`ultracode`
   "finds" them — but only inside *prompt text* and inside tool-call logs of
   this very project; they are not structured fields. Parse JSON keys, never
   grep.) ⇒ The genuine "workflow mode was engaged" signal is the **presence of
   a `subagents/workflows/wf_<id>/` directory** — that, not effort, is what
   distinguishes a real dynamic-workflow run from a plain Task-tool fan-out.
2. **`attributionSkill` and `attributionPlugin` are first-class fields** on
   assistant **usage** lines (verified: 186/186 such lines in session
   `208fad2a` carry both, e.g. `attributionSkill:"superpowers:executing-plans"`,
   `attributionPlugin:"superpowers"`). This is **authoritative** skill/plugin
   attribution — strictly better than the v2.1 `<command-name>` + `Skill`
   tool_use heuristic, and it lets us cost-weight a skill/plugin by the *exact*
   `message.usage` of its own lines instead of the "usage at/after invocation"
   approximation.
3. **New top-level / usage fields now present** (Claude Code ≥ 2.1): top-level
   `entrypoint` (e.g. `claude-vscode`), `permissionMode`, `version`,
   `context_management`; new line `type`s `mode`, `last-prompt`
   (carries the verbatim last user prompt), `queue-operation`,
   `file-history-snapshot`, `attachment`, `system` (hook summaries);
   `message.usage` now also has `service_tier`, `iterations`, `speed`,
   `inference_geo`, `cache_creation` (object).
4. **Why "Claude-model workflows look missing":** Claude sub-agent logs *do*
   exist (haiku/opus appear in the NJU & PolyU `subagents/` dirs). But when a
   native-Claude run uses **ultracode**, the expensive Opus/Fable orchestration
   stays in the **main session log**, and only cheap **haiku** sub-agents (or
   none) write `agent-*.jsonl`. Session `208fad2a` (this very project: 510×
   `claude-fable-5` + 17× `claude-opus-4-8` in the *main* thread) spawned **no
   `subagents/` dir at all**. So the Workflows tab — which groups *sub-agent
   files* — correctly shows the cheap models and can't see the main-thread
   orchestration cost. The fix is in Phase 7 (associate the orchestrating main
   session with its run, and surface a "main-session orchestration" cost line).
5. **The "no prompt samples" on PhD_Python_Figures** is the 30-day
   content-analysis window: aggregates are all-time but `recentPrompts` is
   capped at 30 days, and that project's prompts are older. Phase 9c adds a
   configurable window + an explicit "samples are older than the window" note.

### Phase 8 — Content-analysis calibration (real-total anchoring)

Implements the answer to "can it be more precise than text-length estimates?"
(see also V2.2 spec C8). **Totals are already exact** (`message.usage`); only
the *within-request category split* is estimated. Keep the estimates for
*relative* shares, then **scale them to the real totals**:

- assistant-side categories (`assistantText` / `assistantThinking` /
  `toolCalls`) are scaled so their sum equals the real `Σ output_tokens` of the
  scope;
- input-side categories (`userPrompts` / `toolResults`) scaled to the real
  `Σ (input_tokens + cache_creation_input_tokens)`.

Apply the same anchoring to the per-session **thinking share** (thinking-share
× real output tokens = a defensible "real thinking tokens" figure). Run the
calibration **per day and per week** (cache the factors; recompute on refresh).
Label calibrated numbers distinctly from raw estimates in the UI (e.g. a
"calibrated" badge vs the existing `estimatedNote`). New setting
`claudeCodeUsage.analysis.calibrate` (default **on** — it strictly improves
absolute accuracy and changes no shares).

**Acceptance:** calibrated category token sums equal the exact `usage` totals
for the scope (±rounding); shares are unchanged vs raw; thinking-token absolute
figure agrees with the calibrated output total; toggling the setting off
restores the raw estimates.

### Phase 9 — AI advice UX + **Prompt Optimizer** (⛔ checkpoint; the headline feature)

This is the most on-mission feature in the whole spec ("帮你用得更好"). Three
parts.

**9a. Make the advice entry obvious.** Move the `✨ Get AI Advice` button out of
the Content-tab section header into a **prominent card at the top of the Content
tab** (title + one-line "what this does" + the button + a "Preview demo" link).
Keep the command-palette entry too. Nothing else changes about the existing
report generation.

**9b. Prompt Optimizer (new).** A small always-available box (top of Content
tab, under the advice card) where the user pastes a rough prompt and gets back a
**ready-to-paste, optimised prompt** for Claude Code. It reuses the configured
OpenAI-compatible endpoint (`advice.*`) — i.e. the user's **cheap** model does
the pre-processing, then the polished prompt is handed to the strongest Claude
model to actually execute. Pipeline (all opt-in, surfaced as checkboxes on the
box):
  - **Structure & disambiguate** (always): turn scattered asks into a scoped,
    one-concern-per-block instruction with explicit acceptance criteria —
    exactly the rewrites the advice reports already demonstrate (single-concern
    prompts, CSS-selector/line-number specificity, "locate then confirm before
    editing"). Seed the optimizer's system prompt from the recurring advice
    patterns.
  - **Resolve fuzzy references** (opt-in): for "the bug", "that file", "the 4
    levels" — instruct the cheap model (or a cheap local file scan) to find the
    likely file/symbol/location and inline the concrete path into the prompt, so
    the expensive model doesn't burn context hunting.
  - **Distil long pasted material** (opt-in, user-decided per the report's
    advice): either (a) write the material to a temp `.md` and replace it in the
    prompt with a `cat` instruction (cache-friendly), or (b) insert an explicit
    "extract N points, ≤1 sentence each" distillation instruction. Offer the
    choice; don't silently truncate.
  - **Aesthetic / fuzzy-judgement scaffolding** (opt-in): when the task is a
    vague design ask ("make it prettier"), have the model propose 2–3 named
    style directions (or accept a reference URL/screenshot path and describe its
    style), and fold the chosen direction into concrete CSS/layout asks.
  Output: the rewritten prompt in a read-only box with a **Copy** button (and,
  if a distillation file was produced, its path). No keystrokes are sent to any
  terminal — copy/paste only (same read-only trust boundary as the rest of the
  extension; this only *calls the user's own configured API*, never Claude Code).
  Gate the whole box behind `advice.apiKey` being set (demo text otherwise).

**9c. Window/setting housekeeping for advice quality.** Add
`claudeCodeUsage.advice.promptWindowDays` (default 30) and, when the newest
captured prompt is older than the window, print an explicit note in the report
("no prompt samples within N days — advice is based on aggregates only; widen
`promptWindowDays` or run in an active project") so the PhD_Python "vague
advice" case is self-explaining rather than mysterious.

**Design answers to fold into the help text / README:**
- *Does Claude Code already structure my prompt internally?* It does light
  internal handling, but it does **not** pre-optimise *your* rough wording — the
  Prompt Optimizer is a genuinely new pre-pass, run on your cheap model.
- *Token-saving significance?* The direct token shave is modest; the real
  savings are (i) avoiding failed/repeat runs — the advice flagged a **12×**
  repeated identical instruction as the single biggest waste signal — and (ii)
  offloading cheap lookups/distillation to a cheap model so the expensive model
  starts with a tight, concrete prompt. Frame it as *quality + iteration
  reduction*, not raw token shaving.
- *Do I have to set up sub-agents myself?* No — choosing ultracode/workflow
  **auto-spawns** sub-agents. What you optionally configure is **which model**
  cheap sub-agents use (sub-agent model / CC-Switch routing); the advice's
  "use a cheaper model for simple subagents" refers to that routing, not to
  manually creating agents.

**Acceptance:** advice button is a prominent card; the Prompt Optimizer turns a
pasted rough prompt into a structured one via the configured cheap model, with
the opt-in stages working and a Copy button; everything gated on `advice.apiKey`;
no terminal automation; the older-than-window note appears when applicable.

## 9. Open product question for the maintainer (decide before Phase 9)

The Prompt Optimizer sends the user's *draft prompt* (and possibly a cheap-model
file scan) to the configured API. That's the same trust level as today's advice
feature (user-triggered, user's own key) — but it is now part of the everyday
authoring loop, not an occasional report. Confirm it should stay **manual
copy/paste** (recommended — keeps the read-only boundary) rather than ever
writing into a terminal, and whether the cheap-model **file scan** for fuzzy
references is acceptable (it reads project files the extension can already read,
but now sends snippets to the API). Default to the conservative choice: file
scan **off** unless the user ticks it.
