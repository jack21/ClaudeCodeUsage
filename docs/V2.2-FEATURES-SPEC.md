# V2.2 Implementation Spec — Visualization, Settings Panel & Behaviour Optimisation

> Companion to `V2.2-AUTOMATION-SPEC.md` (AI issue/PR handling). This file covers the *product* side of v2.2: new visualizations, a consolidated settings panel, and behaviour-optimisation features. Same usage pattern: hand to a fresh Fable 5 session — *"Implement docs/V2.2-FEATURES-SPEC.md, phase by phase, stop after Phase A for a checkpoint."* Prereq: v2.1.0 (workflow attribution) shipped — several features below consume the `WorkflowUsage` model and per-skill stats it introduces.
>
> **Guiding constraint (repeat after me): the project is *lightweight and efficient*.** Every feature here must earn its place; default to off or to a collapsed/secondary view; never crowd the primary surfaces. If a feature can be a column instead of a tab, make it a column.
>
> **Data honesty constraint:** the headline token/cost numbers are *exact* (read from each request's `message.usage`). Everything in this spec that *estimates* — model-fit re-pricing, cache-waste, efficiency heuristics — must be visibly labelled as an estimate and must never be mixed into the exact totals.

## Phase A — Settings consolidation (do this FIRST; everything else adds settings)

**Problem:** 18+ flat `claudeCodeUsage.*` settings clutter the VS Code Settings UI. Adding more (this spec adds several) makes it worse.

**Approach:**
1. Keep the VS Code `contributes.configuration` entries (some users edit settings.json directly, and VS Code sync needs them) but **group them** with dotted sub-namespaces so the Settings UI nests them: `display.*` (language, decimalPlaces, compactNumbers, timezone), `refresh.*` (interval, fileWatching, pauseDashboardRefresh), `quota.*` (tracking, warnThreshold…), `advice.*` (already grouped), `analysis.*` (enableContentAnalysis, projectGroupingMode), and a new `experimental.*` namespace (see the note below). Migrate old keys with backward-compatible reads (pattern already used for `advice.apiKey`).
2. Add a **Settings panel inside the dashboard** — a gear/"⚙ Settings" tab that renders the grouped options as a compact form (checkboxes, selects, number inputs), writing through `vscode.workspace.getConfiguration().update` (the same channel the auto-refresh toggle already uses). This is the user-facing "don't make me hunt in VS Code settings" surface.
3. The dashboard form and the VS Code settings stay in sync (both read/write the same config keys).

**Experimental-features convention (introduced here, used by Phase B2 and any future opt-in risk feature):** anything that is not purely read-over-logs, or that we are not yet confident in, lives under `claudeCodeUsage.experimental.*`, defaults to **off**, and renders in the settings panel under a clearly separated "🧪 Experimental" group — each with a one-line risk note and a link to its section in the README. No blanket "enable all experimental" switch; each feature is toggled and explained on its own. This keeps the trust boundary legible: a user can see, in one place, every behaviour that goes beyond passive monitoring.

**Acceptance:** all existing settings reachable from the dashboard panel; changing one there persists and reflects in VS Code settings and vice versa; old flat keys still honoured; experimental settings visibly grouped and off by default.

## Phase B — Pre-emptive quota warning + optional auto-continue

Two related behaviours around the 5-hour window. **High value** — this is the "I hate hitting the limit mid-task" pain.

### B1. Pre-emptive warning (default on, threshold configurable)
- New setting `claudeCodeUsage.quota.warnThreshold` (default **60**; the observed cost of one max-effort ultracode run is ~50–60% of a 5h window).
- When a dashboard refresh sees 5h utilisation cross the threshold **upward**, show a single VS Code notification: "You're at N% of your 5-hour quota. A full ultracode run typically uses ~M% — consider finishing current work before the reset to avoid a mid-run pause." Debounce: at most once per 5h window (track the window's `resets_at`); dismissible.
- Pure status-bar colour already warns ≥80%/≥95%; this adds the actionable *predictive* nudge earlier.

### B2. Optional auto-continue (experimental, default OFF)
- Inspiration & prior art — **credit required:** [AutoContinue](https://github.com/CrisChenYingyan/AutoContinue) by CrisChenYingyan. It avoids losing a long task when the 5h window resets mid-run, and avoids the ~40% re-cost of resuming after a pause (lost prompt cache). Add it to a new **Acknowledgements** section of the README (and every localized README), and link it from this setting's description. Crediting the prior art we build on is a project convention, not an afterthought — see `feedback_prior_art_survey` in spirit: survey first, then credit.
- **Scoping (the trust boundary):** this extension is read-only over Claude Code's data and must *stay* that way by default. Auto-continue means *sending input to Claude Code*, a different trust level — so it lives under `claudeCodeUsage.experimental.autoContinue` (off by default, badged Experimental per Phase A). When the 5h window resets while a session was paused at the limit, surface a **one-click "Resume now"** action (a notification button), not silent keystrokes. Investigate whether Claude Code exposes a resume command the extension can invoke via `vscode.commands.executeCommand`; if not, the one-click action copies the resume command to the clipboard / shows it. Do **not** automate keystrokes into a terminal — too fragile and surprising.
- Document the trade-off plainly in the README, next to the acknowledgement.

**Acceptance:** warning fires once per window above threshold; auto-continue is off by default and clearly experimental; when on, the resume affordance appears only at a real reset; AutoContinue is credited in the README and the setting description.

## Phase C — Behaviour-optimisation analytics

### C1. Skill ROI view (Content tab; consumes v2.1 per-skill stats)
"You have 8 skills installed; this month you used 3. The other 5 cost ~X tokens/turn in loaded context." Cross-reference installed skills (`~/.claude/.claude.json` → `skillUsage`/`pluginUsage`, and the `.claude/skills` / plugin dirs) against observed invocations. Render a "Skill ROI" sub-section: skill · installed? · invocations (30d) · est. tokens · a "rarely used — consider disabling" flag. Actionable framing: disabling a low-value skill beats adding a new one.

### C2. Cost prediction
Dashboard card: "If you run another ultracode workflow now, est. cost ≈ $X (based on your last 10 runs)." Compute from `WorkflowUsage` history: mean + spread of recent workflow costs; show range, not false precision. Also a simpler "this session is trending toward ~$Y by end of day" from today's slope. Keep it one small card, clearly labelled estimate.

### C3. Cumulative usage line chart
A cumulative (running-total) cost line over the selected period, in addition to the per-day bars — makes "how fast am I burning" legible. Reuse the gridded chart infra; a line is a polyline over the same x-axis. Add as a toggle on the existing daily/monthly chart (bars ↔ cumulative line), not a new tab.

### C4. Duration / efficiency stats — and the "一字千金" vanity block
Two layers. The serious layer is for tuning; the fun layer is for sharing (feeds the C6 export).

**Serious (per model, and optionally per task type):** tokens-per-active-minute, output share (output ÷ total tokens), and a rough "efficiency" read (output tokens ÷ cost). Goal: see which model/settings get more done per unit time/cost during autonomous runs. Derive duration from session/workflow start–end timestamps. Surface in the Sessions/Workflows tabs as columns, plus a small per-model summary. Mark efficiency as heuristic.

**Fun / "一字千金" (a small, shareable stats block):** the satisfying single-number framings, computed from the *exact* totals:
- **Cost per output token / per 1k output tokens** — the literal "一字千金" rate ("every thousand output tokens cost you $X").
- **Cost per prompt you typed** — `totalCost ÷ messageCount` (messageCount is user-typed prompts, per the v2.0.2 attribution fix), i.e. "$ per thing you actually asked for."
- **Cost per session** and **most expensive single turn** (the priciest assistant request) — a fun "hall of fame".
- **Cache savings realised** — the *opposite* of waste: `cacheReadTokens × (inputPrice − cacheReadPrice)`, framed as "your prompt cache saved you ≈$N this month." This is a genuinely good-news number and a strong share hook.

Keep the fun block visually compact (a row of stat chips); it is decoration over the real data, never a replacement for it.

### C5. Model + config recommendation (the important one — make it genuinely smart)
**Goal, restated:** not "switch to a cheaper model" but **"match the model and Claude Code settings to what this task actually needed."** Sometimes that means *down*grading an over-powered run; sometimes it means *up*grading a task that thrashed on a too-weak model and cost more in rework than a frontier model would have. The headline line per session/workflow reads like:

> *"This task looks **moderate** difficulty (low thinking share, hit cache well, 4 turns, no rework). An equivalent run on **\<model X\>** at **\<thinking/effort Y\>** would cost ≈ $Z (−W%). Suggested setup: `/model …`, thinking `…`."*

**Pipeline:**
1. **Fingerprint the task — difficulty *and* realised quality**, from the session/workflow:
   - *Difficulty signals:* total output tokens, thinking-token share, tool-call count and diversity, number of turns, peak context, cache-hit rate, ratio of output to input, edit/diff volume.
   - *Quality / smoothness signals (the new part):* did the run converge or thrash? Proxy from turns-to-completion, retry/error records, `[Request interrupted]` markers, repeated edits to the same file, and abandoned tool calls. A task that took many corrective turns is a *quality* signal that the model was under-powered (or mis-configured), independent of token count.
2. **Classify into a difficulty tier** (e.g. trivial / moderate / hard / frontier-grade) from the difficulty signals, and a **smoothness grade** from the quality signals.
3. **Capability-match, both directions:**
   - Over-powered + smooth (frontier model, high effort, low thinking share, trivial tier) → recommend the *cheapest* model/config whose tier still **meets** the difficulty, with the projected saving.
   - Under-powered + thrashy (cheap model, hard tier, many corrective turns) → recommend *upgrading*, and show that the rework likely cost **more** than a frontier run would have (re-price the actual wasted turns). This is the counter-intuitive, high-trust recommendation no cost tool makes.
   - Right-sized → say so, and stop (don't invent a switch).
4. **Recommend Claude Code config, not just the model.** Map the fingerprint to a concrete setup: model selection (`/model`), thinking level (think / think hard / think harder / ultrathink), and reasoning effort for models that support it. E.g. "frontier model but drop effort to medium" for large-output-but-low-reasoning tasks; "keep model, raise thinking" for tasks that thrashed on reasoning.
5. **Re-price the same token profile** under the recommended setup using the existing pricing table (Fable 5 / Opus / Sonnet / Haiku / DeepSeek …) → projected $ delta (saving *or* added cost). Present as an estimate and caveat clearly: different models tokenise and "think" differently (esp. the Opus 4.7+ tokenizer using up to ~35% more tokens), so the re-price is directional, not a quote.
6. **Let the AI-advice layer (v2.1 Phase 5) do the nuanced reasoning** — feed it the fingerprint + smoothness grade + the re-priced table and have it write the one-paragraph recommendation. The deterministic re-pricing bounds it; the LLM adds the judgement.
7. **Surface:** a "Model fit" line per session/workflow + two monthly rollups — *"you'd have saved ≈ $N by right-sizing the \<category\> tasks"* **and** *"you'd likely have avoided ≈ $M of rework by using a stronger model on these K hard tasks."* The two-sided framing is **the headline differentiator** — no other tool does retrospective, quality-aware model-and-config fit.

### C6. One-page export / share — as an image, not just text
**Purpose (explicit):** make usage easy to *share* — both privately and as soft promotion of the extension (think a WeChat-Moments / Xiaohongshu card). So the export's first-class output is an **image**, with Markdown as the text fallback.

- **Share card (PNG, the headline output):** render the dashboard summary — today / month / all-time totals, top models, cost composition, quota snapshot, and the C4 "一字千金" / "cache saved you $N" chips — into a styled, self-contained card. Offer a couple of sizes: a portrait card (~1080×1350, Xiaohongshu/Moments) and a landscape card (~1200×630, link-preview / Twitter). Implementation: the dashboard is already a webview, so build the card as DOM/SVG and rasterise client-side (`SVG → <canvas> → canvas.toDataURL('image/png')`), then hand the data URI back to the extension to write via `vscode.window.showSaveDialog`. No new runtime deps; no external service.
- **Markdown export:** the same summary as a self-contained `.md` (for READMEs, issues, changellogs).
- **"Link output" — be honest about privacy:** a true shareable *URL* would require uploading the data somewhere, which violates the project's local-only/read-only posture. So default to **no upload**: the share is the PNG/MD file the user saves, plus a "Copy summary" action (Markdown to clipboard). If a hosted link is ever added, it must be an explicit, separate opt-in (Phase A experimental namespace), never the default.
- **Shareability hygiene:** strip/abbreviate absolute project paths, and let the user toggle whether dollar amounts are shown (some will want to share the *shape* of usage, not the spend). A small "🔗 Promote" footer crediting the extension is fine on the card and doubles as soft marketing.

### C7. Cache-churn diagnostics (NEW — the "why is cache-miss so high?" feature)
**Problem you reported:** sometimes the **cache-creation ("Input Cache (Miss)") tokens are very high**, and it tends to happen after **switching models** or at the **start of a new long prompt**. That intuition is correct and worth turning into a feature.

**The mechanism (document this in the feature's help text):** the prompt cache is **per-model** and has a short (~5-minute) TTL. Any of these forces a fresh **cache write** (billed at ~1.25× input) instead of a cheap **cache read** (~0.1× input) on the next request:
- switching models mid-session (the new model can't read the old model's cache),
- idling longer than the cache TTL (the cache expires),
- a large new prompt / context reset (new prefix to cache).
A workflow that churns the cache pays repeatedly to *re-establish* context that a steadier workflow would have read cheaply.

**The feature (a "Cache health" sub-section in the Content tab):**
- **Ratios & money:** per session and per day, cache-read vs cache-write token ratio, the $ spent on cache writes, and an estimate of the $ "wasted" re-creating cache that was discarded shortly before. Label the waste figure an estimate.
- **Event detection, straight from the logs we already read** (no new permissions, stays read-only):
  - **Model switches** — when `message.model` changes between consecutive requests in one session, flag it: "model switch here — cache flushed, +X write tokens (~$Y)."
  - **TTL gaps** — when the timestamp gap between consecutive requests exceeds ~5 min, flag "idle past cache TTL — cache likely expired."
  - **Big new prompts** — the first request of a large fresh context, where a write is expected (so we *don't* cry wolf about unavoidable writes).
- **Tie into AI advice:** feed the advice model the session's model-switch pattern + cache-write spikes, plus (optionally, read-only) the configured default model / CC-Switch presence from `~/.claude.json`. Note: the **model-switch pattern is already fully observable from the logs** (the per-request `message.model` field), so the core advice needs no extra config access — reading settings only adds "you have CC Switch configured to flip models often." Example advice: *"You switched models 6× in this session; each switch discards the prompt cache (≈Xk tokens re-created, ≈$Y). Batch same-model work, or finish a task before switching models, to keep cache reads cheap."*
- **Caveat:** we infer cache behaviour from token deltas and timestamps; we can't see Anthropic's cache internals, so present every churn attribution as a heuristic.

## Phase D — Rendering fixes carried in alongside

- **Cache-miss bar invisible in *both* the 2026 themes** (reported): the "Input Cache (Miss)" / cache-creation bar shows its number but **no visible bar** in **Dark 2026 *and* Light 2026** (not dark-only — in Light 2026 the light purple washes out on white; in Dark 2026 the dark purple collapses into near-black). Root cause: the bar colour leans on `--vscode-charts-purple` / pink gradients that have no guaranteed contrast against these themes' backgrounds. Fix: give the bar a **theme-independent, min-contrast fill plus a subtle 1px border** so it is always visible regardless of theme, and a **min-width** so even a tiny value renders a sliver next to the number. Verify across Dark+, Dark Modern, Light+, Light Modern, **Dark 2026, Light 2026**, and a high-contrast theme. (If a quick, certain fix surfaces during a v2.0.x patch it can ship earlier — it's a pure CSS change.)
